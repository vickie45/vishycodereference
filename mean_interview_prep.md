📚 MEAN Stack Basics & Core Concepts

At the core of the MEAN stack is a unified, JavaScript-based architecture. These fundamentals are where most interviews will begin.

Topic Key Points for Your Answer
What is the MEAN Stack? A collection of four JavaScript-based technologies: MongoDB (NoSQL database), Express.js (back-end web framework), Angular (front-end framework), and Node.js (server-side runtime). It allows developers to build dynamic web applications using a single language, JavaScript, across the entire stack.
Why Use MEAN Over LAMP? The key differentiator is the unified use of JavaScript from client to server. This eliminates context-switching between languages (e.g., PHP and JS), simplifies development, and promotes code reusability. It's also well-suited for building modern, real-time, single-page applications (SPAs).
Advantages of MEAN Full-Stack JavaScript: One language for everything. JSON Everywhere: Data flows seamlessly in its native format. Scalable & Flexible: Especially with MongoDB's schema-less design and Node.js's non-blocking I/O. Cost-Effective: The entire stack is open-source.
TypeScript in Angular While JavaScript is the stack's foundation, Angular uses TypeScript. TypeScript is a superset of JavaScript that adds optional static typing. It supports core Object-Oriented Programming principles like encapsulation, inheritance, polymorphism, and abstraction, which helps in building large-scale, maintainable applications.

🗄️ MongoDB: Data Modeling & Performance

MongoDB is the database layer. For a senior role, you're expected to know more than just basic CRUD; you need to think in terms of data modeling, performance, and scalability.

MongoDB Basics

Topic Key Points for Your Answer
Documents & Collections A document is the basic unit of data, similar to a row in SQL, but stored as a JSON-like object (BSON). A collection is a group of documents, analogous to a table in a relational database. Crucially, documents within the same collection can have different fields (a flexible schema).
Why BSON? BSON (Binary JSON) is a binary-encoded serialization of JSON-like documents. MongoDB uses it because it provides extensions that JSON lacks, such as support for more data types (e.g., Date, ObjectId, binary data) and is more efficient for both storage space and traversal speed.
Indexing Purpose Indexes in MongoDB support the efficient execution of queries. Without indexes, MongoDB must perform a collection scan—reading every document—to find matching results. Indexes store a small portion of the data in an easy-to-traverse form, dramatically speeding up read operations, though they can slow down writes and consume storage.
When Not to Use Node.js Node.js is not ideal for CPU-intensive tasks. Its single-threaded event loop can be blocked by heavy computation, making the application unresponsive. For these scenarios, consider offloading tasks to worker threads, separate microservices, or using a different technology.

🔷 Advanced MongoDB: Aggregation & Schema Design

Topic Explanation & Scenario
Aggregation Pipeline A framework for data processing that consists of multiple stages. Each stage transforms the documents as they pass through the pipeline. Think of it as an assembly line for your data. Scenario: Calculate total revenue per month from an orders collection. Solution: Use a pipeline with: 1) $match to filter orders by a date range, 2) $project to extract the month and year, 3) $group to sum the totalAmount for each month, and 4) $sort to order the results chronologically.
$lookup (Joins) This stage performs a left outer join with another collection, allowing you to combine data from multiple sources in a single query. It's the MongoDB equivalent of a SQL JOIN. Scenario: Fetch all users along with their recent orders. Solution: Use $lookup on the users collection, specifying the orders collection as "from," userId as "localField," and _id as "foreignField". You can then use $unwind to flatten the resulting array of orders if needed.
Data Modeling: Embedding vs. Referencing This is a core architectural decision. Embedding (denormalization): Storing related data within a single document. Scenario: Storing comments directly within a blog post document. Benefits: Faster reads with a single query. Drawbacks: The document can grow large (up to 16MB), and updating a single comment requires updating the entire post document. Referencing (normalization): Storing references (IDs) to documents in other collections. Scenario: Storing user IDs in an orders collection. Benefits: Data is more normalized, smaller documents, and easier to manage relationships. Drawbacks: Requires multiple queries or $lookup to fetch related data. The choice depends on data access patterns, read/write ratio, and the size of the related data.
Optimizing Slow Queries 1. Analyze with explain(): Use the explain() method to understand the query plan. Identify if a COLLSCAN (collection scan) is happening instead of an IXSCAN (index scan). 2. Create Proper Indexes: Build single-field, compound, or text indexes based on your query patterns. 3. Review Schema Design: Overly large or deeply nested documents can hurt performance. Consider refactoring to a referenced model. 4. Limit Returned Fields: Use projections to only fetch the data you need.
Handling Duplicate Records Scenario: A user collection has duplicate email addresses. Solution: 1. Use an aggregation pipeline to $group documents by the email field, counting occurrences. 2. $match to find groups with a count > 1. 3. For each duplicate group, keep the first document (e.g., by _id) and mark the rest for deletion. 4. Perform a deleteMany operation on the marked documents. 5. Prevention: Create a unique index on the email field to prevent future duplicates.

⚙️ Express.js: Middleware, Routing, and Security

Express.js is the back-end framework. At 5 years of experience, you should be an expert in middleware patterns and security best practices.

Express.js Fundamentals

Topic Key Points for Your Answer
Role of Express.js Express is a minimalist web framework for Node.js that simplifies building web applications and RESTful APIs. It provides a robust set of features for web and mobile applications, including routing, middleware support, and a simpler interface for handling HTTP requests and responses.
What is Middleware? Middleware functions are functions that have access to the request object (req), the response object (res), and the next function in the application’s request-response cycle. They can execute any code, make changes to the request and response objects, end the request-response cycle, or call the next middleware in the stack.
next() Function The next() function is a callback that passes control to the next middleware function in the stack. If next() is not called within a middleware function, the request will be left hanging and never sent to the client, leading to a timeout.
Types of Middleware Application-level: Bound to the app object (e.g., app.use()). Router-level: Bound to an instance of express.Router(). Error-handling: Defined with four arguments: (err, req, res, next). Built-in: Like express.json() and express.static(). Third-party: Like cors, helmet, morgan.
Route vs. Query Parameters Route parameters: Used to identify a specific resource, like /users/:id. Accessed via req.params.id. Query parameters: Used to filter or sort data, like /users?age=25. Accessed via req.query.age. Route params are part of the URL path, while query params are in the URL after ?.

🔷 Advanced Express.js: Security & Error Handling

Topic Explanation & Scenario
JWT Authentication Flow JWT (JSON Web Token) is a stateless method for authentication. Flow: 1. User logs in with credentials. 2. Server validates credentials and generates a JWT containing user info (payload), signed with a secret key. 3. Server sends the token to the client. 4. Client stores the token (e.g., in localStorage or HTTP-only cookie) and sends it in the Authorization: Bearer  header with each subsequent request. 5. Server middleware verifies the token's signature and extracts the user payload, making it available for the rest of the request cycle.
Protecting Routes Scenario: Only authenticated users can access /dashboard. Solution: Create a custom middleware function called protect. This middleware: 1. Extracts the token from the Authorization header. 2. Verifies the token using jwt.verify() and your secret key. 3. If valid, attaches the decoded user object to req.user and calls next(). 4. If invalid or missing, it sends a 401 Unauthorized response.
Global Error Handling Scenario: Centralize error handling for a clean and consistent API. Solution: Use Express's built-in error-handling middleware. You create a function with four parameters: (err, req, res, next). When you pass an error to next(err) in any route or middleware, Express will skip all remaining non-error middleware and jump directly to this error handler. This allows you to send a standardized error response and log errors in one place.
Securing an Express API A multi-layered approach: 1. Use Helmet: Helps secure your app by setting various HTTP headers (e.g., XSS protection, disabling X-Powered-By). 2. Enable CORS: Configure Cross-Origin Resource Sharing to control which domains can access your API. 3. Implement Rate Limiting: Prevent brute-force and DoS attacks by limiting requests from a single IP address. 4. Validate Input: Sanitize and validate all user inputs on the server-side to prevent injection attacks. 5. Use Environment Variables: Never hardcode secrets or API keys. Use a package like dotenv.
Logging Every Request Scenario: Track all incoming API requests for debugging and analytics. Solution: Create a simple application-level middleware and place it at the very top of your middleware stack. app.use((req, res, next) => { console.log(${req.method} ${req.url} - ${new Date().toISOString()}); next(); });

🖥️ Angular: Components, RxJS, and State Management

Angular is the front-end framework. For a 5-year role, you need to demonstrate a deep understanding of reactive programming and state management.

Angular Core Concepts

Topic Key Points for Your Answer
Purpose of Angular Angular is a platform and framework for building single-page client applications using HTML and TypeScript. It implements core functionality as a set of TypeScript libraries that you import into your apps, offering features like two-way data binding, dependency injection, and a powerful component-based architecture.
Dependency Injection (DI) DI is a design pattern in which a class requests dependencies from external sources rather than creating them itself. In Angular, the DI framework provides declared dependencies to a class when that class is instantiated. This makes components more modular, easier to test, and less coupled to their dependencies.
Scope in JavaScript Scope refers to the context in which variables are accessible. Global Scope: Variables declared outside any function are accessible everywhere. Local Scope: Variables declared within a function are only accessible inside that function. Block Scope: With let and const, variables are only accessible within the block {} they are defined in. Understanding scope is crucial for avoiding variable naming conflicts and memory leaks.
Routing Guard A routing guard is an interface in Angular that allows you to control whether a user can navigate to or away from a route. You can implement guards like CanActivate (to allow navigation to a route), CanDeactivate (to warn about unsaved changes), and Resolve (to pre-fetch data).

🔷 Advanced Angular: RxJS, State, and Performance

Topic Explanation & Scenario
RxJS: Observable, Subject, BehaviorSubject Observable: A representation of any set of values over any amount of time. It's lazy; it won't start emitting values until you subscribe to it. Subject: A special type of Observable that is also an Observer. You can push values to it, and all its subscribers will receive them. It doesn't have an initial value. BehaviorSubject: A type of Subject that requires an initial value and emits its current value to any new subscribers. This is ideal for sharing state, like the current user's authentication status. Scenario: Communicate between sibling components. Solution: Use a shared service with a BehaviorSubject. One component can call .next(data) to update the value, and other components can subscribe to the BehaviorSubject.asObservable() to react to changes.
When to Use NgRx Scenario: You have a large application with complex, shared state. Use NgRx When: You have a lot of shared state, complex state transitions, and you need a single source of truth and predictable state management. It's excellent for time-travel debugging and enforcing a unidirectional data flow. Don't Use NgRx When: Your application is small to medium with simple, isolated state. For these cases, a simple service with a BehaviorSubject is often sufficient and reduces unnecessary boilerplate.
NgRx Core Concepts Store: A single, immutable data structure that holds the application's global state. Actions: Plain objects that describe unique events (e.g., [Login Page] Login). They are dispatched to the store. Reducers: Pure functions that take the current state and an action, and return a new state without mutating the old one. Effects: Handle side effects like API calls. They listen for actions, perform an asynchronous task, and then dispatch new actions with the result (success or failure).
Performance Optimization Lazy Loading: Load feature modules (and their associated JavaScript bundles) only when the user navigates to their routes. This drastically reduces the initial bundle size and speeds up the application's first load. OnPush Change Detection: Use ChangeDetectionStrategy.OnPush to tell Angular to only check a component for changes when its @Input properties change, an event originates from the component, or an Observable it's subscribed to emits. This can significantly reduce the number of change detection cycles. trackBy in *ngFor: Use trackBy to help Angular identify which items in a list have changed, been added, or been removed. Without it, Angular will re-render the entire list, which can be a major performance bottleneck for large collections.
Implementing Search with Debouncing Scenario: Build a real-time search feature that calls an API as the user types. Solution: In your component, create a FormControl for the search input. In ngOnInit, subscribe to its valueChanges observable. Pipe the observable through debounceTime(300) (to wait for a 300ms pause in typing), distinctUntilChanged() (to ignore if the new value is the same as the last), and then switchMap(searchTerm => this.apiService.search(searchTerm)). This ensures you only make an API call after the user has stopped typing, avoiding unnecessary network requests.

🚀 Node.js: Performance, Scalability & Architecture

Node.js is the runtime. At the senior level, you're expected to understand its inner workings and how to build scalable, resilient applications.

Node.js Fundamentals

Topic Key Points for Your Answer
Role of Node.js Node.js is a JavaScript runtime built on Chrome's V8 JavaScript engine. It allows developers to run JavaScript on the server, outside of a browser. It's designed for building scalable network applications.
Callback Function A callback is a function passed as an argument to another function, which is then invoked (or "called back") after the completion of an asynchronous operation. This pattern is fundamental to Node.js's non-blocking I/O model, allowing it to handle multiple concurrent operations without waiting for any single one to finish.
Buffer Class Buffers are used to handle streams of binary data. They are a global class in Node.js, so you don't need to require a module to use it. Buffers act like arrays of integers but represent raw memory allocations outside the V8 heap. They're essential when dealing with file systems, TCP streams, or any operation involving raw binary data.
Control Stream Flow "Control flow" or "control stream work" in Node.js refers to managing the execution order of asynchronous operations. Since Node.js is non-blocking, you need patterns like callbacks, Promises (async/await), or streams to ensure tasks happen in the correct sequence and to handle errors gracefully.

🔷 Advanced Node.js: Event Loop, Scaling, and Performance

Topic Explanation & Scenario
The Event Loop The event loop is the core of Node.js's non-blocking I/O architecture. It's a single-threaded, semi-infinite loop that picks up and executes callbacks from different queues in a specific order. The phases are: 1. Timers: Executes setTimeout/setInterval callbacks. 2. Pending Callbacks: Executes I/O callbacks deferred to the next loop iteration. 3. Idle, Prepare: Used internally. 4. Poll: Retrieves new I/O events. 5. Check: Executes setImmediate() callbacks. 6. Close Callbacks: Executes close event callbacks. Microtasks: process.nextTick() and Promises are executed after the current operation, before moving to the next phase. process.nextTick() has a higher priority than Promises.
Handling CPU-Intensive Tasks Problem: A CPU-heavy task (e.g., image processing, complex calculation) will block the event loop, making the server unresponsive. Solution 1: Worker Threads: Run the CPU-bound code in a separate thread, preventing it from blocking the main event loop. Solution 2: Child Processes: Fork a new Node.js process to handle the task, communicating via IPC (Inter-Process Communication). Solution 3: Offload to a Queue: Use a message broker (like Bull or RabbitMQ) to enqueue the task and have a separate worker process consume it.
Clustering & Scaling Clustering: The cluster module allows you to create multiple worker processes that all share the same server port. This leverages multi-core CPU systems, allowing you to handle more load. The master process manages the workers and distributes incoming connections. child_process is a more general module for spawning and communicating with system commands or other Node.js scripts. Horizontal vs. Vertical Scaling: Vertical scaling means adding more power (CPU, RAM) to your existing server. Horizontal scaling means adding more servers to your pool of resources. Node.js is designed for horizontal scaling.
Debugging a Memory Leak Scenario: Your Node.js application's memory usage grows steadily over time. Solution: 1. Use --inspect: Run your app with node --inspect app.js and open Chrome DevTools. Use the Memory tab to take heap snapshots at different times and compare them to find what objects are accumulating. 2. Generate Heap Dumps: Use the heapdump module or process.writeHeapSnapshot() to generate snapshots programmatically. 3. Analyze with Tools: Tools like clinic.js doctor can provide a detailed analysis and pinpoint bottlenecks and memory issues. Common causes include global variables, forgotten timers, and event listeners not being removed.
